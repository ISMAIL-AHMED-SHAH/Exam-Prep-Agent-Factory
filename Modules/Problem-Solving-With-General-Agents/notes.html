Intro

Yeh Course Kis Baare Mein Hai

AI Prompting course mein aap ne seekha AI se baat karna. Tool courses mein seekha tools (plan mode, skills). Ye course sikhata hai teesri layer: in tools se asal masla achi tarah solve karna. Do students ko same assignment milta hai — Student A 25 minute mein clean, verified summary bana leta hai. Student B 90 minute waste kar deta hai, conversation messy ho jati hai, AI mistakes karta hai, restart karna parta hai. Same AI, same task — farq sirf itna: Student A ne 7 principles follow kiye, Student B ko pata nahi thay.

Mode 1 vs Mode 2:
  • Mode 1 — Problem Solving: tool kholo, task solve karo, result ship karo. Yehi is course ka focus hai.
  • Mode 2 — Building AI Workers: permanent AI assistants banana jo khud chalte rahein (alag course).
Foundation

Lindy Effect — Purane Tools Kyun Important Hain

Computing ke sab se important tools sab se purane bhi hain: terminal, files, Git, SQL. Decades se hain aur ab bhi kaam karte hain. Lindy Effect kehta hai: jo cheez lambe arse se useful hai, wo aage bhi useful rahegi.

Key Points:
  • AI apna naya tareeqa nahi banata — wohi purane proven tools use karta hai (terminal commands, files, git, SQL)
  • Aapka role change hota hai, khatam nahi hota: aap problem define karte ho, result verify karte ho
  • AI in purane tools ko outdated nahi karta — unhein zyada valuable banata hai
PRINCIPLE 1 Bash is the Key
⚠ Failure it prevents: "Agent only talks, doesn't act"

Chatbot sirf sawalon ka jawab deta hai. Ye tools action lete hain: files parhte hain, documents likhte hain, commands chalate hain. Bash us black-screen terminal ki zubaan hai. Cowork/OpenWork mein same idea step-cards ke through dikhti hai.

Beginner mistake:
  • "How should I organize my notes?" → sirf advice milegi, kuch organize nahi hoga
  • Fix: specific instruction do jo ek finished file produce kare, na ke paragraph of suggestions
⚠️ Mental model: Agent ke "hands" hain. Hands ko brief karo, brain ko nahi — method mat batao, sirf outcome batao.
PRINCIPLE 2 Code as Universal Interface
⚠ Failure it prevents: "Prose request keeps getting misread"

Do parts hain: (1) jab task ek command se zyada complex ho, AI code likh kar solve karta hai. (2) Format utna hi zaroori hai jitna khud request — plain paragraph vague hota hai, table/checklist/schema clear hota hai.

Bash vs Code:
  • Bash (P1) = Hands — navigate, search, move, one command at a time
  • Code (P2) = Brain — compute, transform, orchestrate, persist, integrate
5 Powers jo Code deta hai:
  • Precise thinking (exact numbers, not rough guesses)
  • Workflow orchestration (multi-step rules ek sath run hoti hain)
  • Organized memory (folders/files se progress save)
  • Universal compatibility (alag formats ek sath combine)
  • Instant tool creation (jo app exist nahi karti, AI khud bana deta hai)
⚠️ Aapke do kaam hamesha rehte hain: (1) Problem clearly define karna (format batao, words nahi), (2) Result verify karna (code parh sako, ya claims source se match karo).
PRINCIPLE 3 Verification as a Core Step
⚠ Failure it prevents: "Output looks right but breaks in production"

Finished-looking output verified output nahi hota. Models plausible cheezein banate hain, jo correct se different hai. Verification workflow ka ek step honi chahiye, afterthought nahi.

Key Rule:
  • Jis agent ne output banaya, wohi uska sab se kharab verifier hai — same blind spots hote hain
  • Verification ko independent path chahiye: apni reading, alag model, test, type-checker, ya database constraint
  • "Is this correct?" khud usi AI se poochna real verification nahi — wo tarafdari karega
⚠️ Verification prompt pattern: Har factual claim list karo → source location dhoondo aur exact quote lo → jo claim ground na ho, flag karo → jab tak sab flags resolve na hon, save mat karo.
PRINCIPLE 4 Small, Reversible Decomposition
⚠ Failure it prevents: "One big change nuked an afternoon"

Bara task chote steps mein torro. Ek step khatam karo → check karo sahi hai → save karo → agla step. 12-step task ek message mein doge to AI step 5 tak drift kar jata hai.

Rule of Thumb:
  • Agar kisi change ko reverse karne mein 2 minute se zyada lagega, to change bara tha
  • Pixar lesson: 1998 mein Toy Story 2 ki files accidentally delete hui — sirf ek employee ki personal backup ne film bachai. Saving progress "yaad rakhne wali cheez" nahi, process mein built-in honi chahiye
  • Undo trap: git reset --hard sab kuch last save point tak erase karta hai — agar tumne save (commit) nahi kiya, wo gaya
⚠️ Enforcement prompt: Task ko sab se chote steps mein todo. Har step ke baad: (1) dikhao kya kiya, (2) verification check chalao, (3) commit/save karo, (4) agla step shuru karne se pehle meri OK ka wait karo.
PRINCIPLE 5 Persisting State in Files
⚠ Failure it prevents: "Agent forgets what we decided yesterday"

Conversation band karte hi AI sab bhool jata hai. Fix: important info ko file mein save karo. Sab se important file rules file hai — Claude Code/Cowork mein CLAUDE.md, OpenCode/OpenWork mein AGENTS.md. AI har naya session shuru hote hi ye automatically parhta hai.

Key Points:
  • Run /init — first draft khud ban jata hai
  • Length: first draft under 250 words, weeks baad under 60 lines. 500 words se zyada = documentation ban gayi, bohat lambi
  • Har line isliye ho ke kuch mistake hui thi uske bina
  • Shape: What this is → Where things live → Critical rules → On-demand references
  • Plan files: multi-session tasks ke liye — "Read plans/q4-launch.md aur step 4 se continue karo"
  • Jab files kaam na dein (repeat questions har baar naya code chahiye), database use karo (jaise Neon)
⚠️ Hierarchy: Conversation = volatile. Project folder files = durable. Referenced files = on-demand.
PRINCIPLE 6 Constraints and Safety
⚠ Failure it prevents: "Agent touched files I didn't authorize"

Limits problem nahi hain — yehi AI ko safe banate hain. Agar AI kuch bhi bina pooche kar sakta hai, to har second watch karna parega. Limits set karne se AI slow nahi hota — balke aap AI ko zyada freedom de sakte ho bharosay se.

3 Universal Trust Levers:
  • Scope — kaunsi files/folders AI dekh sakta hai
  • Connections — kaunsi external services tak pohanch sakta hai
  • Approvals — kab AI aapki OK ka wait karta hai
Autonomy Ladder (5 rungs):
  • 1. Watching closely → 2. Ambient supervision → 3. Walk away → 4. Act without asking → 5. Scheduled/automated
⚠️ Prompt-injection trap: Bahar se aayi files (email, PDF, resume) mein hidden instructions ho sakti hain. Outside files ke sath "watching closely" level pe raho. Agar AI ka plan aisi files/services mention kare jo tumne nahi mangi, approve mat karo.
PRINCIPLE 7 Observability
⚠ Failure it prevents: "Don't know what the agent actually did"

Sirf wohi direct kar sakte ho jo dekh sakte ho. Agent ka har meaningful action tumhein real-time ke qareeb visible hona chahiye. Ek novel task pe kam se kam ek dafa execution view zaroor dekho — "agent ne kuch aisa kiya jo expect nahi tha" ka sab se bara reason: user ne dekha hi nahi tha.

5 Symptoms — Session Off the Rails:
  • Chat ke unrelated purane parts reference karta hai
  • Responses lambe aur vague ho jate hain, hedging barh jati hai
  • Kisi purani constraint se contradict karta hai
  • Baar baar apologize karta hai bina progress ke
  • Aisi files/folders/connectors chhoone ka propose karta hai jo mention nahi hui thin
⚠️ Jab ye symptoms dikhein: Typing rok do. Aur prompt se fix karne ki koshish mat karo — /clear chalao ya naya session kholo, sirf zaroori 1-2 facts paste karo, wahin se continue karo. Reset almost hamesha rescue se faster hota hai.
PART 2 The Four-Phase Workflow

7 principles production mein ek 4-phase loop mein simat jate hain. Loop haath mein aa jaye to principles phases ke andar khud-ba-khud fire hote hain.

4 Phases:
  • 1. Explore (Bash + Observability) — relevant files parho, unknowns samne lao. Read-only, koi writes nahi
  • 2. Plan (Code-as-Interface + Persistence) — structured artifact ke tor par likha hua plan banao, save karo. Ye sab se important phase hai — zyada tar leverage yahin hai
  • 3. Implement (Decomposition + Verification) — plan ko chote atomic steps mein execute karo, har step ke baad verify karo, commit/save karo
  • 4. Commit (Observability) — final verification pass, decisions ko rules file mein wapas save karo agli baar ke liye
⚠️ Constraints (P6) sab ko wrap karta hai: ye ek phase nahi, poore loop ke around ki boundary hai — scope, connectors, approval mode session start pe set hote hain aur har phase ko govern karte hain.
PART 2 5 Failure Patterns

Jab loop mein kuch ghalat ho, wo taqreeban hamesha in 5 named patterns mein se ek hota hai. Pattern pehchanna batata hai kaunsa principle use karna hai.

PART 3 A Worked Example

Task family: ek complex incoming artifact review karo, jo matter karta hai wo identify karo, verified claims ke sath structured response produce karo.

Do tracks, same shape:
  • Engineer track: contractor ka pull request — diff review, risks flag, response likho
  • Domain-expert track: vendor ka master services agreement — redline standard se deviations flag, comparison memo
4 Phases is task pe:
  • Explore → PR diff ya MSA parho, read-only summary save karo
  • Plan → structured review/redline plan banao (severity HIGH/MED/LOW ke sath), approval ke liye pause karo
  • Implement → plan ko chote steps mein execute karo, har concern verify karo
  • Commit → final check, rules file update karo agli baar ke liye
⚠️ Note: Domain alag ho sakta hai (PR ho ya MSA), lekin workflow shape identical rehti hai — yehi is principle ko portable banata hai.

📝 One-Line Recap — Sab Concepts

  1. Lindy Effect: purane proven tools (terminal, files, git, SQL) AI ke sath aur zyada valuable ban jate hain
  2. P1 — Bash is the Key: AI ko doer treat karo, advisor nahi; hands ko brief karo
  3. P2 — Code as Interface: complex task ke liye code, aur format utna hi zaroori jitna content
  4. P3 — Verification: output ko independent path se verify karo, khud se nahi
  5. P4 — Small Reversible Decomposition: bara task chote steps mein, har step ke baad checkpoint
  6. P5 — Persisting State in Files: rules file (CLAUDE.md/AGENTS.md) mein important decisions save karo
  7. P6 — Constraints and Safety: scope + connections + approvals set karo, autonomy ladder dheere climb karo
  8. P7 — Observability: execution view novel task pe zaroor dekho, 5 symptoms pe reset karo
  9. 4-Phase Workflow: Explore → Plan → Implement → Commit, Constraints (P6) sab ko wrap karta hai
  10. 5 Failure Patterns: Drift, Confident Wrong, Big Bang, Scope Creep, Black Box — har ek ka apna fixing principle
  11. Worked example: domain chahe jo bhi ho, workflow shape same rehti hai — yehi portability hai
