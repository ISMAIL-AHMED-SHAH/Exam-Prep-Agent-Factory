📝 Markdown In, HTML Out

Roman Urdu Notes · 14 Concepts · 4 Parts · GIAIC Agent Factory

Intro

Do Document Languages

Markdown plain text hai jisme thoray extra marks structure dikhate hain (# heading ke liye, - list item ke liye). HTML wo language hai jis se web pages banti hain. Dono programming nahi hain. Skill ye hai: kaunsa format kis direction mein use karna hai.

Power users ka split:
  • AI ko Markdown mein likho — structure guessing khatam kar deta hai
  • AI se HTML mein jawab mango — designed page hi wo lamba jawab hai jo koi busy insaan actually parhta hai
⚠️ Asal sawal jo har format decision settle karta hai: "Ye document sab se aakhir mein kaun parhega?" Insaan browser mein → HTML. AI (ya future chat) → Markdown. Social feed → plain text.
PART 1 The Two Languages
Concept 1

Why Agents Need Structure

Ek admin ne agent ko prose mein likha "younger kids ko afternoon heat mein mat rakhna" — agent ne phir bhi under-8 races 2pm pe schedule kar din, kyunke constraint sentence ke andar dabi hui thi. Jab usne ek alag ## Hard constraints heading banaya, agent ne perfectly follow kiya.

Key Rule:
  • Constraint ki information nahi badli — sirf structure mein uski visibility badli
  • Jo cheez agent ko galat nahi karni chahiye, wo apni alag heading ya bullet mein honi chahiye, kabhi sentence ke andar nahi
Concept 2

The Asymmetry: Markdown In, HTML Out

Pehle AI hamesha Markdown mein jawab deta tha. Ab Anthropic ke Thariq Shihipar ke mutabiq: lambi Markdown unreadable hoti hai (100 lines ke baad log parhna chhor dete hain), uski ceiling low hai (sirf headings/lists/tables), jabke HTML ki ceiling almost na-mumkin hai (styled tables, diagrams, interactivity) aur wo shareable hai (link click, kuch install nahi karna).

DirectionFormatWajah
Aap → AgentMarkdownStructure ambiguity hatati hai
Agent → AapHTMLRich, readable, shareable
Agent → AgentMarkdownCompact, precise context
⚠️ "HTML out" ki ek condition hai — reader insaan ho. Naya chat kuch yaad nahi rakhta, isliye aaj ke notes jo kal ke fresh chat mein paste karoge, wo bhi "agent → agent" hain, chahe dono taraf insaan hi ho. Confusion ho to Markdown chuno.
PART 2 Markdown, the Writing Language
Concept 3

Headings: AI Ko Batao Kya Important Hai

# heading banata hai, zyada # deeper level. Headings AI ko batati hain kya important hai aur kaunsa detail kis section ke neeche aata hai.

3 Rules jo 90% mistakes rokte hain:
  • Ek document mein sirf ek # title — do titles matlab do documents fuse ho gaye
  • Level skip mat karo (# se seedha ###) — tree toot jata hai
  • Headings ko claims banao, labels nahi — "Budget" nahi, "Budget: PKR 50,000 hard ceiling"
Concept 4

Lists: Bullets for Sets, Numbers for Sequences

Bullets (-) kehte hain: ye ek set hai, order matter nahi karta, har item independently true hai. Numbers (1.) kehte hain: ye ek sequence hai, order hi point hai — step 3, step 2 assume karta hai.

Key Rule:
  • Numbered list do to agent order ko instruction ka hissa samajhta hai — kabhi step 4 se step 1 se pehle shuru nahi karega
  • Checkable-bullet habit: "Page fast honi chahiye" checkable nahi. "Page 3G pe 3 second mein load ho" checkable hai
Concept 5

Code Blocks: Triple Backticks

Triple backticks (```) AI ko batate hain "ye data hai, instruction nahi." Bina fence ke, agent har cheez ko instruction samajh kar follow karne ki koshish karta hai — jaise ek error message mein "delete all rows? (y/n)" likha ho aur agent usay execute karne lage.

Sab se useful trick:
  • Sirf format describe mat karo — ek example row fence mein dikhao. Concrete example ko misread karna mushkil hai, written description ko easy
Concept 6

Links and Images

Prompt mein link do to AI usay actually visit kar ke parh sakta hai — apni memory se guess karne ke bajaye. Image ke liye ![description] — brackets ke andar description hi wo cheez hai jo AI dekhte waqt "parhta" hai, ek caption ki tarah jo batati hai kis pe dhyan dena hai.

Concept 7

Your First Complete Specification

Poora spec skeleton: Goal → Context → Requirements → Hard constraints → Out of scope → Expected output. Do sections sab se zyada kaam karte hain: Out of scope (enthusiastic over-delivery rokta hai) aur Expected output (format drift rokta hai).

Sab se important habit:
  • Spec ko execute karne se pehle validate karo: agent se poocho ambiguities list karo, missing constraints batao, spec ko 10 mein se grade karo
  • 2-3 rounds mein spec 6 se 9 tak jati hai — ye sab se sasti quality improvement hai, kyunke har downstream artifact spec ki clarity inherit karta hai
⚠️ Mindset shift: Specification "kaam se pehle ka paperwork" nahi — ye khud asal kaam hai. Spec improve karo, har regeneration improve ho jati hai.
PART 3 HTML, the Reading Language
Concept 8

Why Ask for HTML at All

HTML 5 cheezein add karta hai: Information density (tables, diagrams, ASCII art ke bajaye), Navigability (tabs, table of contents), Shareability (link click, install nahi), Interactivity (sliders, toggles), aur "copy back" loop (settings text ke tor par wapas mil jati hain).

⚠️ Honest caveat: HTML Markdown se zyada tokens use karta hai — 3-line answer ke liye pure overhead hai. Trade tab worth hai jab output lamba, visual, shared, ya revisited ho. Quick chat back-and-forth mein plain response hi sahi default hai.
Concept 9

You Don't Write HTML; You Prompt For It

Trigger word ("HTML artifact") easy hai. Asal skill brief mein hai. "Ek page banao" kehna "khana banao" kehne jaisa hai — kuch to milega, lekin jo chahiye tha wo nahi.

Har HTML request mein 4 cheezein:
  • Kaun parhega — ek co-founder laptop pe ek dafa, ya reference jo baar baar khulega?
  • Kya hona chahiye — specific pieces naam se batao (diagram, table, summary strip)
  • Interactive honi chahiye ya nahi — sliders? copy button? tabs?
  • Kaise parhi jayegi — "ek dafa parhna" vs "reference" vs "client ko forward karna"
Concept 10

The Five HTML Patterns

#PatternKab Use Karein
1Plans & ExplorationsDecision se pehle multiple options grid mein compare karna
2Explainers & ReportsLamba document ek-page HTML summary mein badalna
3Code ReviewColor-coded diffs, annotated code
4Design PrototypesJo words mein describe karna mushkil ho (color, roundness) — live sliders

Through-line:
  • Har pattern mein: Markdown/data andar jata hai, HTML bahar aata hai, aur jo conversation continue karna ho wo text ke tor par wapas aata hai
Concept 11

When the Destination Is a Feed, Not a Browser

Social feed browser nahi hai. WhatsApp, LinkedIn, Facebook aapki formatting ignore karte hain — sirf plain text dikhate hain. Raw HTML paste karna hamesha ghalat hai (angle brackets ki wall ban jati hai).

Key Points:
  • WhatsApp: sirf *bold* aur _italic_ chalta hai, baaki sab ignore
  • LinkedIn/Facebook: plain text, pehli line attention pakarni chahiye (LinkedIn "…see more" ke peeche baaki chupa deta hai)
  • HTML sirf 2 jagah matter karta hai: link preview card (Open Graph tags se) aur designed images jo HTML se bana kar PNG save ki jayein

Concept 13

When the Destination Is a Document, Not a Browser

Sawal ab thora aage barhta hai: "kaun parhega, aur wo is se karega kya?" Sign karna, print, edit, present, ya numbers pe kaam — har action apna format mangta hai.

Format by Action:
  • Sirf parhna → HTML link
  • Sign/print/file → PDF
  • Edit/comment → Word (.docx)
  • Present → Slides (.pptx)
  • Numbers pe kaam → Excel (.xlsx)
  • Doosre tool mein feed → CSV
⚠️ Parallel: "CSV data ke liye wohi hai jo Markdown text ke liye hai" — plain, machine-facing. "Excel data ke liye wohi hai jo HTML text ke liye hai" — visual, human-facing.
PART 4 One Exercise, Three Motions
Concept 14

The Three Motions

Same skill, teen tareeqon se apply hoti hai: Chat (Claude.ai/ChatGPT/Gemini — jo paste karo usi pe kaam), Terminal (Claude Code/OpenCode — poore folder ki files khud parh leta hai), Desktop (Cowork/OpenWork — visual app, plan dikhata hai aur approval ka wait karta hai).

Kaunsa Motion Kab:
  • Notes ek paste/attachment mein fit hon → Chat
  • Notes bikhri hui hon multiple files/folders mein → Terminal
  • Visual app chahiye jo touch karne se pehle poochay → Desktop
⚠️ Desktop tools ke sath hamesha: "pehle plan dikhao, jab tak main 'go' na kahoon kuch mat likho." Ye plan-then-approve habit zaroori hai jab bhi AI tool real files touch kar sakta ho.

📝 One-Line Recap — Sab Concepts

  1. Structure requests headings/bullets se karo — jo galat nahi hona chahiye, apni heading/bullet mein rakho
  2. Asymmetry: aap → agent Markdown, agent → aap HTML, agent → agent Markdown. Sawal: "kaun parhega last?"
  3. Headings batati hain kya important hai — ek title, no skipped levels, claims banao labels nahi
  4. Bullets = set (order matter nahi), Numbers = sequence (order hi point hai)
  5. Triple backticks: "ye data hai, instruction nahi" — example row show karo, sirf describe mat karo
  6. Links AI ko real page parhne dete hain; image caption wahi hai jo AI "dekhta" hai
  7. Spec skeleton: Goal → Context → Requirements → Hard constraints → Out of scope → Expected output. Execute se pehle validate karo
  8. HTML 5 cheezein deta hai: density, navigability, shareability, interactivity, copy-back loop — lekin short answers ke liye overhead hai
  9. HTML mangte waqt 4 batao: kaun parhega, kya hona chahiye, interactive ya nahi, kaise parha jayega
  10. 5 Patterns: Plans/Explorations, Explainers/Reports, Code Review, Design Prototypes, Throwaway Editors
  11. Social feed browser nahi — plain text post karo, HTML sirf preview card aur image ke liye
  12. Publishing: Claude.ai publish (fastest) → GitHub Gist → GitHub Pages → Netlify (permanent options)
  13. Document destination format se decide hoti hai: PDF sign/file, Word edit, Slides present, Excel numbers, CSV machine
  14. 3 Motions: Chat (paste), Terminal (files khud parhta hai), Desktop (plan-then-approve)
